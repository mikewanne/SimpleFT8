## 1. State-Klassen-Analyse (3 Gruppen)

### Gruppe A: state ∈ (IDLE, TIMEOUT)
- Pendings: alle None / leer
- `start_qso` einfach durchlaufen
- **Kein Bug-Risiko**

### Gruppe B: state ∈ (CQ_CALLING, CQ_WAIT)
- `cq_mode = True`
- `_pending_reply` evtl. gesetzt (jemand hat unser CQ beantwortet)
- `_caller_queue` evtl. nicht leer (mehrere Antworter)
- `_pending_hunt_reply`, `_pending_rr73` IMMER None (wir sind nicht im Hunt)
- **`_on_station_clicked`-Pfad:** `stop_cq()` aufrufen, dann `start_qso()`.
  ⚠️ `_pending_reply` und `_caller_queue` werden NICHT resetet!
- **CQ_WAIT-Skip in `start_qso` Z.241:** `if state not in (IDLE, CQ_WAIT)` —
  CQ_WAIT wird KEIN State-Reset gemacht (kein _set_state(IDLE) aufgerufen).
  Bei CQ_CALLING wird Reset gemacht.

### Gruppe C: state ∈ (TX_CALL, WAIT_REPORT, TX_REPORT, WAIT_RR73, TX_RR73,
                       WAIT_73, TX_73_COURTESY)
- `cq_mode = False` (oder True wenn `_was_cq=True` aus CQ-Resume)
- `_pending_reply` evtl. gesetzt (CQ-Modus war aktiv vor dem Hunt)
- `_pending_hunt_reply` evtl. gesetzt (Antwort auf unseren Call ist im Slot)
- `_pending_rr73` evtl. gesetzt (RR73 wartet auf naechsten TX-Slot)
- `_caller_queue` evtl. nicht leer
- `qso.courtesy_73_sent` evtl. True (wenn state == TX_73_COURTESY)
- **`_on_station_clicked`-Pfad:** kein CQ-Stop noetig. `start_qso()`
  setzt `_set_state(IDLE)` + neue QSOData. ⚠️ Pendings bleiben!

## 2. Encoder-State bei Stations-Klick

**Encoder-Felder die relevant sind:**
- `is_transmitting` (Audio laeuft) → Klick wird zurueckgewiesen (Z.68)
- `tx_even` (Slot-Paritaet) → in Z.91 ueberschrieben fuer neuen Klick
- `audio_freq_hz` → wird im naechsten Encode aus encoder-State gelesen
- `_pending_message`, `_replace_message`, `_replace_lock` → P1.9 Replace-Logik

**Race-Window:**
1. Encoder-Worker schlaeft `wait` bis `_WAKE` (Z.146-157 decoder.py-aehnlich
   in encoder.py)
2. Wake → preEncode → Audio-Buffer-Generierung
3. `is_transmitting` wird True (mw_qso.py:_on_tx_started Pfad)
4. PTT on → Audio-Send via VITA-49 (BLOCKING ~13s)
5. Audio fertig → `is_transmitting` False

**Klick-Window-Risiken:**
- A) Vor Encoder-Wake: `is_transmitting=False` → Klick akzeptiert. tx_even
  wird ueberschrieben. Encoder wacht spaeter mit neuem `tx_even`. **OK.**
- B) Wake → preEncode (~50ms): `is_transmitting=False` aber Encode
  laeuft. tx_even kann ueberschrieben werden — Encoder hat aber bereits
  alten Wert kopiert in Worker-Local. **Race, aber kein Bug** (alte Audio
  wird gesendet, neuer QSO startet im naechsten Slot).
- C) Nach `is_transmitting=True`: Klick zurueckgewiesen. **OK.**
- D) Nach Audio-Ende: wie A. **OK.**

→ Encoder-State ist KEIN Bug-Risiko fuer P1.14.

## 4. `_pending_reply` + `_was_cq` Interaktion (KP-Kandidat)

**Szenario:** CQ_WAIT, jemand hat geantwortet → `_pending_reply = caller_A`.
Mike klickt aber Station-B.

**Aktueller Code:**
```python
_cq_was_active = self.qso_sm.cq_mode  # True
if _cq_was_active:
    self.qso_sm.stop_cq()              # cq_mode = False
    self.control_panel.set_cq_active(False)
...
self.qso_sm.start_qso(B)
# start_qso Z.241: state == CQ_WAIT → SKIP _set_state(IDLE)!
#                  state-Wechsel via _set_state(TX_CALL) am Ende
# qso = neue QSOData(B)
# _pending_reply BLEIBT auf caller_A
if _cq_was_active:
    self.qso_sm._was_cq = True         # OK, fuer Resume
```

**Folge:**
- state == TX_CALL fuer B
- `_pending_reply = caller_A`
- Naechster Slot: `_send_cq()` wird NICHT mehr aufgerufen (cq_mode=False)
- ABER: irgendein Code-Pfad koennte `_pending_reply` lesen und auf A
  springen. **Pruefen!**

→ **Pendings RESET noetig** in `start_qso` wenn state != IDLE.

## 6. `_caller_queue` — Designfrage

**Aktueller Code:** `_caller_queue` wird bei `start_qso` NICHT geleert.

**Szenario:** CQ_WAIT mit 3 Caller in Queue. Mike klickt Station-B (nicht
in Queue). `start_qso(B)` startet QSO mit B. Queue bleibt mit 3 Caller.

**Frage:** Was soll passieren?
- A) Queue clear (clean break, B war Mike's Wahl)
- B) Queue erhalten (nach QSO-B-Ende und CQ-Resume werden sie abgearbeitet)
- C) Queue erhalten WENN `_was_cq=True` (CQ-Resume nach QSO), sonst clear

→ Designentscheidung. **V2-Empfehlung: Option B** (Mike will durchgaengig
durcharbeiten, das ist ja Sinn der Queue). R1 soll bestaetigen.
