## Kritik des Prompts „P48 — DT-System aufräumen + tunen (V2)“

Nachfolgende Findings orientieren sich am definierten Scope (Single-Operator, FlexRadio, Mike-only) und an KISS. Overengineering wird explizit benannt.

| Schwere | Finding | Datei:Zeile | Empfehlung |
|:--------|:--------|:------------|:-----------|
| **Bug** | **Schnell-Konvergenz wird nie feuern wegen falscher Initialisierungs-Logik** — AK6 prüft `_is_initial` und `_cycle_count == 1`. Aber nach AK2/AK3 wird `_is_initial` beim Kaltstart *immer* auf `False` gesetzt: Der Hardware-Default (0.26) wird in `_load_for_current_key` zurückgegeben → `set_mode`/`set_band` setzen `_is_initial = (saved_val == 0.0)` → `False`. Damit ist die Bedingung `_is_initial and _cycle_count == 1` nie erfüllt. Der optimierte Pfad existiert nur auf dem Papier. | `core/ntp_time.py:set_mode/set_band` + `update_from_decoded` | Entweder `_is_initial` auf `True` setzen *trotz* Hardware-Default (z. B. neues Flag `_hardware_fallback_used`), oder `_is_initial` über `_hardware_default_offset != 0.0` entscheiden lassen. Der aktuelle Prompt widerspricht sich selbst. |
| **Risiko** | **Test T11 bildet Realität nicht ab** — T11 setzt `_hardware_default_offset = 0.0` und erwartet Schnell-Konvergenz. Im Produktivbetrieb ist der Default `0.26`. Dadurch wird ein Pfad getestet, der im echten System *so nie durchlaufen wird*. Der Test ist wertlos, wenn nicht beide Defaultszenarien abgedeckt sind. | `tests/test_p48_dt_optimization.py:test_fast_convergence_threshold` | Zusätzlichen Test mit `_hardware_default_offset = 0.26` schreiben, der zeigt, dass die Schnell-Konvergenz auch mit nicht-null-Initialwert korrekt arbeitet (oder den Bug aus Finding 1 fixt und dann testet). |
| **Over-engineering** | **`_SIBLING_PRIORITY`-Dict ist zu komplex für 3 Modi** — FT8, FT4, FT2 mit maximal zwei Fallback-Stufen. Ein einfaches `if-elif` in `_load_for_current_key` wäre lesbarer und vermeidet die Gefahr von Tippfehlern in Schlüsseln. Das Dict fügt eine Indirektionsebene, die keinen messbaren Wartungsvorteil bringt (Mike-only, Hobby-Tool). | `core/ntp_time.py: Top-Level` | Ersetzen durch: `FT2 → try FT4 → try FT8`; `FT4 → try FT8`; `FT8 → keine`. Das sind 3 Zeilen statt 7 + Dict-Definition. KISS. |
| **Risiko** | **Fallback-Kette für FT2 unvollständig** — Laut AK3 sucht FT2 zuerst FT4, dann FT8. Aber wenn auf demselben Band *weder* FT4 *noch* FT8 gespeichert sind, greift der Hardware-Default. Das ist korrekt. **Aber**: Wenn FT4 einen gespeicherten Wert hat, FT8 aber nicht, wird der FT4-Wert genommen. Das kann schiefgehen, weil die Median-Basis für FT4 kleiner ist. Der Prompt begründet die Reihenfolge mit „FT8 hat die meisten Stationen → verlässlichster Median". Dann müsste FT2 *zuerst* auf FT8 fallen, nicht zuerst auf FT4. | `core/ntp_time.py:_SIBLING_PRIORITY` | Reihenfolge ändern: `FT2 → [FT8, FT4]`. FT8-Daten sind robuster als FT4-Daten (10k vs ~100 Einträge laut Hintergrund). Die aktuelle Reihenfolge priorisiert den weniger verlässlichen Median. |
| **Verbesserung** | **Tests decken nicht den Fall „FT2 hat eigenen Wert, FT4+FT8 existieren" ab** — T7 testet, dass der eigene Wert Vorrang hat (OK). Es fehlt: FT4 ist gespeichert, FT2 hat *keinen* Wert, FT8 hat auch keinen → Erwartung: Fallback auf FT4. (Der Fall „FT2 hat eigenen Wert" ist trivial.) | `tests/test_p48_dt_optimization.py` | Ergänzung z. B. `test_cross_mode_ft2_falls_back_to_ft4_when_ft4_saved_ft8_missing`. |
| **Hinweis** | **Schnell-Konvergenz: `valid`-Liste wird mehrfach allokiert** — In AK6 wird `valid` einmal für `len(valid)` und einmal für `statistics.stdev(valid)` verwendet. Das ist in Ordnung. **Aber**: `statistics.stdev(valid)` wird *nur* im Fast-Path berechnet. Die Funktion hat aktuell schon `valid = [dt for dt in dt_values if -2.0 <= dt <= 2.0]` weiter oben. Die zusätzliche Bedingung `len(valid) >= 10` ist redundant zum `_FAST_CONVERGENCE_MIN_STATIONS`-Check. | `core/ntp_time.py:update_from_decoded` | `_FAST_CONVERGENCE_MIN_STATIONS` könnte entfallen und stattdessen `len(valid) >= 10` direkt verwenden (Konstante ist Overkill). Oder die Konstante belassen, aber dann ist der `len(valid) >= 10`-Kommentar irreführend. Einheitlichkeit. |
| **Risiko** | **`statistics.stdev` bei `len(valid) == 10` kann bei stark ausreißenden Werten > 0.1 sein, aber das wird durch `_FAST_CONVERGENCE_MAX_STDEV = 0.1` abgefangen** — korrekt. **Aber**: Der Prompt sagt „bei FT4/FT2 selten“. Wenn FT4/FT2 nur 1-2 Stationen haben, wird der Fast-Path nie feuern → OK. **Allerdings**: Der Fast-Path prüft `_is_initial`, was nach Finding 1 nie `True` ist. |
| **Hinweis** | **`_WAKE_OFFSETS` ist out-of-scope, aber der Prompt erwähnt es im Kommentar als `# P48-C: Hardware-Default ...`** — das ist ein Copy/Paste-Überbleibsel. Der Block `# P48-C:` in `_load_for_current_key` referenziert das falsche AK. | `core/ntp_time.py: Kommentarzeile` | `# P48-C` → `# P48-C + P48-B: ...` korrigieren, da dort beide Features implementiert werden. |
| **Verbesserung** | **Test T11/T12/T13 nutzen `fresh_ntp` mit `_is_initial = True` — aber nach Real-Logik (Finding 1) wäre `_is_initial = False`** — die Tests testen einen Zustand, der im Produktivbetrieb *nie* eintritt (solange der Hardware-Default > 0.0 ist). | `tests/test_p48_dt_optimization.py: test_fast_*` | Entweder die Test-Erwartungen anpassen (wenn der Bug gefixt ist) oder einen expliziten Test für den Fall „_is_initial=True trotz Hardware-Default" schreiben. |
| **Risiko** | **Kein Test für den Fall, dass `set_hardware_default` *nicht* aufgerufen wird** — der Default `0.0` ist für Test-Umgebungen gedacht. Wenn im Produktivbetrieb durch einen Init-Reihenfolgefehler `set_hardware_default` nie aufgerufen wird (z. B. weil `_init_core_components` vor dem Settings-Load ausgeführt wird), landen alle Kaltstarts bei 0.0 statt 0.26. Der Prompt setzt voraus, dass der Aufruf immer erfolgt, hat aber keinen Guard/Assertion. | `ui/main_window.py:_init_core_components` + `core/ntp_time.py` | Entweder `_hardware_default_offset` in `_load_for_current_key` mit `settings.rx_hardware_offset_default_s` direkt (statt Setter) initialisieren, oder eine Assertion in `_load_for_current_key` einbauen, die warnt wenn der Wert noch 0.0 ist und der Setter noch nie gerufen wurde (nur im Produktivbetrieb). |

### Zusammenfassung der schwerwiegendsten Punkte

1. **Bug (rot):** Schnell-Konvergenz wird durch die Initialisierungslogik ausgehebelt – der Hauptnutzen von P48-D entfällt.
2. **Risiko (orange):** Cross-Modus-Reihenfolge für FT2 sollte FT8 vor FT4 priorisieren, sonst wird der unzuverlässigere Median benutzt.
3. **Overengineering (gelb):** `_SIBLING_PRIORITY`-Dict ist für 3 Modi überdimensioniert – KISS verletzt.
4. **Inkonsistenz (orange):** Tests bilden die reale Default-Situation nicht ab – sie geben fälschlich grün, obwohl der getestete Pfad im Betrieb nie durchlaufen wird.
