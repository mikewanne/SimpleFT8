## 0. Symptom (Mike-Originalton)

> „Wenn bei mir aktueller CQ ruf läuft und ich eine station die auch CQ
> ruft anklicke müsste mein ruf eigendlich abgebrochen werden"

Erweitert (Field-Test-Erfahrung): bei Stations-Klick waehrend laufendem
QSO oder CQ wird neuer QSO-Versuch gestartet, **aber alter QSO-Status
nicht sauber resetet**. Folge: Geister-Pendings, doppelte Targets,
inkonsistenter State.

**Klick-Szenarien die problematisch sein koennten:**
1. Klick waehrend `state == CQ_CALLING/CQ_WAIT` (CQ laeuft) — bekannt
2. Klick waehrend `state == TX_CALL/WAIT_REPORT/WAIT_RR73/...` (Hunt-QSO laeuft)
3. Klick waehrend `_pending_reply` gesetzt ist (jemand antwortete CQ
   gerade, wir wollen aber andere Station rufen)
4. Klick waehrend `_pending_hunt_reply` gesetzt ist (Antwort auf
   unseren Hunt-Call ist im Slot, wir klicken trotzdem andere Station)
5. Klick waehrend `is_transmitting` (Audio laeuft) — wird derzeit
   einfach ignoriert ohne User-Feedback

    if self.state not in (QSOState.IDLE, QSOState.CQ_WAIT):
        # Laufendes QSO abbrechen → neues starten

    self._was_cq = self.cq_mode

    self.qso = QSOData(                              # neue QSOData
        their_call=their_call,
        ...
    )
    ...
    self._set_state(QSOState.TX_CALL)

def cancel(self):

## 2. Bug-Hypothesen

### 🔴 H1: `start_qso` bei laufendem QSO setzt NICHT alle Pendings zurück

**Befund:** `start_qso` Z.241-245 ruft nur `_set_state(IDLE)` wenn ein
laufendes QSO abgebrochen wird. Aber:
- `_pending_reply` (jemand hat unser CQ beantwortet) — nicht resetet
- `_pending_hunt_reply` (jemand hat unsere Hunt-Antwort gegeben) — nicht resetet
- `_pending_rr73` (laufendes RR73-Senden) — nicht resetet
- `_caller_queue` (Liste anstehender CQ-Antworter) — nicht geleert

**Folge:** Wenn Mike waehrend CQ_WAIT klickt und im Slot davor jemand
antwortete, ist `_pending_reply` gesetzt. Beim Klick auf ANDERE Station:
- `start_qso` setzt state zu TX_CALL fuer NEUE Station
- Aber `_pending_reply` zeigt auf ALTE Station
- Bei naechstem `_send_cq()` oder `on_message_received` → komische
  Effekte (z.B. doppelter Trigger fuer alte Station)

**Reproduktions-Idee:**
1. CQ ruft, jemand antwortet → `_pending_reply = caller`
2. Bevor wir auf den antworten, sehen wir interessantere Station (DX)
3. Klick auf DX → `start_qso(DX)` → state=TX_CALL fuer DX, aber
   `_pending_reply` zeigt noch auf ersten caller
4. Naechster Slot: Decoder feuert message_decoded fuer ersten caller →
   StateMachine triggert moeglicherweise alte Logik

### 🔴 H2: `_active_qso_targets` haelt alte their_call

**Befund:** Z.84 fuegt `msg.caller` (NEUE Station) zum Set hinzu,
**entfernt aber NICHT die alte their_call**. Der Set ist 150s Aging-Liste
fuer aktive QSO-Partner — Decoder behaelt deren Stations-Daten laenger
in `_diversity_stations`.

**Folge:** Bei mehreren schnellen Stations-Wechseln waechst der Set
aber alte Eintraege werden nicht entfernt. Decoder behaelt zu viele
Stations laenger als noetig — kein direkter Bug, aber Memory-Bloat.

### 🟡 H3: Klick waehrend TX silent ignoriert

**Befund:** Z.68-70 `if is_transmitting: return` ohne User-Feedback.
Mike klickt, sieht nichts passiert. Frust-Punkt.

**Folge:** UX-Problem. Mike erwartet sofortiges Feedback („Klick verstanden,
warte auf TX-Ende"). Aktuell stiller Skip.

### 🟡 H4: `_pending_hunt_reply` Geister-Eintrag nach Stations-Wechsel

**Befund:** Mike ruft Station-A (Hunt). Antwort kommt zu spaet (Slot
schon vorbei) → `_pending_hunt_reply = A` ist gesetzt. Mike sieht
inzwischen Station-B im Panel und klickt B.

`start_qso(B)` setzt neues qso fuer B aber `_pending_hunt_reply = A`
bleibt. Naechster Slot: A's Antwort kommt durch Decoder →
`_pending_hunt_reply` wird verarbeitet → Verwirrung.

### 🟢 H5: Encoder-tx_even-Konflikt

**Befund:** Z.91 `self.encoder.tx_even = not their_even` — wenn waehrend
TX-Audio fuer Station-A gerade laeuft und Mike B klickt, wird `tx_even`
ueberschrieben aber `is_transmitting` haette das schon abgefangen.

Aber: zwischen `is_transmitting=False` und `audio_started` gibt es einen
Window von ~50ms wo Encoder schon Re-Encode macht. Race-Condition?

## 3. Reproduktions-Strategie

### Fuer V2/R1 wichtig:
- Mike-Field-Test-Szenarien gemeinsam mit Code-Pfaden mappen
- Welche Pendings sind in welchem State realistisch gesetzt?
- Welche Reset-Logik fehlt konkret?

### Test-Szenarien fuer Plan:
1. **CQ → Klick auf andere Station:** state CQ_CALLING/CQ_WAIT, evtl.
   `_pending_reply` gesetzt, `_caller_queue` evtl. nicht leer
2. **Hunt-QSO → Klick auf neue Station:** state TX_CALL/WAIT_REPORT/etc,
   `_pending_hunt_reply` evtl. gesetzt
3. **CQ + bekannte Station antwortet → Klick auf andere:** `_pending_reply`
   gesetzt, Caller-Queue evtl. nicht leer
4. **TX-Audio laeuft → Klick:** Klick muss verschoben werden bis TX-Ende
   (oder klar signalisiert „warten")
5. **Schneller Doppel-Klick auf 2 verschiedene Stationen:** beide
   Stations-Wechsel innerhalb 1 Slot
6. **Klick auf Station die uns selbst ruft (im RX-Panel-CQ-Sektion):**
   `_pending_reply.target == self.my_call` — sollte Reply-Pfad nehmen,
   nicht neuen QSO
