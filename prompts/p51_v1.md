# P51 — V1 (Gain-Messung vereinheitlichen: 1 Messung, 2 Auswertungen)

> Stand: 14.05.2026 spätnachmittags, nach Bundle J (v0.97.27). Tests 1235.

## 1. Ziel

Aktuell läuft die 18-Zyklus-Gain-Messung pro Scoring-Modus separat — wenn
Mike um 10:00 Standard misst und 5 Minuten später auf DX wechselt, muss er
**komplett neu einmessen** (90 Sek Block). Beide Stores
(`presets_standard.json` + `presets_dx.json`) können divergieren weil sie
zu unterschiedlichen Zeitpunkten unter unterschiedlichen Bedingungen
gemessen wurden (Beispiel 14.05.: 20m FT8 Std 10/10 vs DX 20/10).

**Architektur-Befund (Code-Analyse 14.05.):** Die 18-Zyklus-Roh-Messung
in `ui/dx_tune_dialog.py` ist **scoring-unabhängig**:
- `_phase_data: dict[(ant, gain), list[float]]` hält pro
  (Antenne, Gain-Stufe) SNR-Werte aller dekodierter Nachrichten.
- Nur `_finish()` (Z.356-393) wertet scoring-spezifisch aus:
  - `use_snr=True` (DX): pro Antenne die Gain-Stufe mit bestem Top-5-SNR.
  - `use_snr=False` (Standard): pro Antenne die Gain-Stufe mit den
    meisten Stationen.

→ **Aus identischen Rohdaten ergeben sich zwei unterschiedliche optimale
Gain-Sätze** — beide aus 1 Messung ableitbar.

**Wunschverhalten:**

| Aktuell | Geplant |
|---|---|
| 1 Messung → 1 Auswertung → 1 Store (Std ODER DX) | 1 Messung → **2 Auswertungen** → **beide Stores** |
| Modus-Wechsel: 90-Sek-Neu-Messung | Modus-Wechsel: **instant** (Store ist gültig) |
| Std/DX-Werte aus unterschiedlichen Mess-Situationen | Beide aus derselben Mess-Situation |

**Mike-Begründung:** 50% weniger Mess-Zeit im Alltag, konsistente Werte
über Modi-Wechsel, weniger Friktion.

## 2. Akzeptanzkriterien

**AC1 — DXTuneDialog rechnet beide Auswertungen parallel:**
- `_finish()` in `ui/dx_tune_dialog.py:356-393` produziert NICHT mehr 1
  `_results`-dict sondern **2 Sätze** in einem `_results`-dict:
  ```python
  self._results = {
      "standard": {
          "ant1_gain": int, "ant2_gain": int,
          "ant1_avg": float, "ant2_avg": float,
          "best_ant": "ANT1"|"ANT2", "best_gain": int,
      },
      "dx": {
          "ant1_gain": int, "ant2_gain": int,
          "ant1_avg": float, "ant2_avg": float,
          "best_ant": "ANT1"|"ANT2", "best_gain": int,
      },
      # Top-Level: scoring-aktive Variante fuer abwaertskompatibilitaet
      "ant1_gain": int, "ant2_gain": int, "best_ant": str,
      "best_gain": int, "ant1_avg": float, "ant2_avg": float,
  }
  ```
- Die Top-Level-Felder spiegeln das **aktive scoring_mode** (für
  bestehende Aufrufer in mw_radio die nur 1 Satz erwarten).
- Die `set_rx_antenna` / `set_rfgain`-Aufrufe in `_finish()` nutzen den
  aktiven Modus (= Top-Level-`best_ant`/`best_gain`).
- Per-Antenne-Helper `_best_for(ant, use_snr) -> tuple[int, float]`
  extrahiert die scoring-spezifische Gain+Avg-Logik (DRY).

**AC2 — mw_radio speichert in BEIDE Stores:**
- `_on_dx_tune_accepted` in `ui/mw_radio.py:1424-1455`:
  - Liest `r["standard"]` und `r["dx"]` aus `dialog.get_results()`.
  - Schreibt `_standard_store.save_gain(...)` mit den Standard-Werten.
  - Schreibt `_dx_store.save_gain(...)` mit den DX-Werten.
  - Beide Calls in `try/except` umfasst — wenn EIN Store-Write fehlschlägt,
    läuft der andere durch (Fallback-Sicherheit).
- Die `set_rfgain`-Anwendung am Radio nutzt weiter den aktiven Modus
  (`_gain_scoring_mode`) — also `r["standard"]["best_gain"]` oder
  `r["dx"]["best_gain"]`.

**AC3 — Dialog-Anzeige während Messung:**
- `step_label`/`detail_label` zeigen während Messung: „Messung läuft
  (gilt für beide Modi)" statt scoring-spezifisch.
- Am Ende zeigt der Dialog **beide** Ergebnis-Sätze in `_update_results_display`:
  ```
  ANT1 Gain 10 dB:  Ø +5.3 dB  (18 St.)  ←  (Standard)
  ANT1 Gain 20 dB:  Ø +8.1 dB  (12 St.)  ←  (DX)
  ```
  Markierung `←` zeigt scoring-spezifisch (mit Annotation).

**AC4 — Settings-Kompatibilität:**
- `settings.save_dx_preset(scoring="standard"|"dx")` wird WEITER aufgerufen
  in `_on_dx_tune_accepted` (Z.1457-1468) — **aber zweimal**, einmal pro
  scoring. Setzt sichert Abwärtskompatibilität für Code der direkt aus
  Settings liest.

**AC5 — Mode-Wechsel mid-session ist instant:**
- Wenn `_standard_store.is_valid_gain()` UND `_dx_store.is_valid_gain()`
  beide True (innerhalb 6h-Frist), wird beim Wechsel Std↔DX **keine neue
  Messung** mehr getriggert.
- Bestehende Logik in `_open_dx_tune_dialog` / `_check_diversity_preset`
  bleibt — neu ist nur: nach 1 Messung sind beide Stores gleichzeitig
  valid, daher trigger der Mess-Pfad nicht mehr.

**AC6 — Migration bestehender Werte:**
- Mike's aktuelle Werte (Std 10/10 + DX 20/10) bleiben gültig bis zur
  nächsten Messung. Bei nächster P51-Messung werden **beide** Stores
  überschrieben → divergierter Zustand verschwindet automatisch.
- Kein explizites Reset / Migration-Tool nötig.

- `tests/test_p51_unified_gain.py` NEU: 8-10 Tests siehe Sektion 6.

## 4. Randbedingungen

- **PySide6:** `Signal`/`Slot`.
- **Hardware-Pflicht:** ANT1=TX immer. `_finish` Z.398-400 ruft
  `set_rx_antenna(ant)` (beste-Antenne), `set_rfgain(gain)`,
  `set_tx_antenna("ANT1")` — letzteres bleibt unverändert (TX-Pflicht).
- **Thread-Safety:** Mess-Datenpfad ist single-threaded (Cycle-Signals im
  GUI-Thread). `_phase_data` wird nur in `on_messages_received` (GUI-Thread)
  geschrieben + in `_finish` (GUI-Thread) gelesen. Keine neuen Locks.
- **PresetStore-API:** `save_gain` ist atomar (Memory + Disk), returnt
  `False` bei Disk-Fehler statt Exception. Try-Except in mw_radio kann
  beide Returns prüfen + ggf. Status-Bar-Hinweis loggen.
- **Adaptive-Stop (Z.270-318):** Bleibt scoring-spezifisch — die Schwellen
  (Δ_SNR ≥ 4 dB ODER Δ_STAT ≥ 50%) gelten für **eine** Auswertung. Wenn
  scoring "snr" stop-fähig ist aber "stations" nicht, was tun?
  → V1-Vorschlag: **OR-Logik** — stop wenn EINE Auswertung stop-fähig.
  Falls beide getrennt geprüft werden müssen, R1-Diskussion.
- **`_gain_scoring_mode`:** bleibt als Attribut für UI-Anzeige + Radio-
  Anwendung. Wird NICHT entfernt. Mike's Frage „Toggle entfällt?" → V1:
  **Toggle bleibt** weil er entscheidet welcher Werte-Satz aktiv am Radio
  ausgespielt wird (Std hat kleinere Gain → leiser, DX hat höhere → mehr
  Pegel). Toggle ändert künftig nur **Anwendung**, nicht Messung.
- **Reversibilität:** Wenn Code crashed während Doppel-Save zwischen
  Standard-save_gain und DX-save_gain → 1 Store hat neue Werte, 1 alte.
  Beide unabhängig valid (eigene gain_timestamps), kein konsistenz-bruch.
